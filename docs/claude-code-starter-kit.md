# Claude Code スターターキット（Opus効率運用・統合版）

CLAUDE.md本体とセットで使う付属ファイル群とプロンプト集。
新プロジェクト開始時は「1. 初期セットアップ」を一度だけ実行すれば以後のセッションは低コストで回る。

このキットは3層でコンテキストを管理する:
- **常時ロード**: `CLAUDE.md`（薄く保つ。毎セッション・トークンを消費するため）
- **条件付きロード**: `.claude/rules/*.md`（paths指定。該当ファイルを触るときだけロード）
- **状態**: `docs/STATUS.md`（変化する情報。規約とは分離して化石化を防ぐ）

---

## 1. 初期セットアップ（プロジェクト開始時に一度だけ）

```
mkdir -p docs .claude/agents .claude/rules
```

同梱の CLAUDE.md / docs/*.md / .claude/agents/*.md / .claude/rules/*.md を配置し、
各ファイルのプレースホルダをプロジェクトの実態で埋める。

配置後の最初のセッションで必ず `/cost` を確認し、探索・テストが
実際にhaiku/sonnetへ流れているか検証すること（後述）。

---

## 2. ファイル構成

```
CLAUDE.md                     # 薄いインデックス（常時ロード）
docs/
  STATUS.md                   # 現在地・次の一手（毎セッション更新）
  decisions.md                # 軽量ADR（設計判断の記録）
.claude/
  agents/
    explorer.md               # haiku: 探索・grep（読み取り専用）
    implementer.md            # sonnet: 承認済み方針の実装
    test-runner.md            # haiku: テスト実行・要約
  rules/
    python-tests.md           # paths付き: テストファイル編集時のみロード
```

---

## 3. サブエージェントについて（コスト削減の主レバー）

Opusのメインセッションから安価なモデルへ作業を逃がす。これがコスト効率の要。

- モデルルーティングには既知の不具合報告があるため、導入後に一度 `/cost` で
  実際にsonnet/haikuへ流れているか確認すること。
- 確実性を上げたい場合は環境変数 `CLAUDE_CODE_SUBAGENT_MODEL` を併用
  （環境変数名は最新の公式ドキュメントで綴りを確認してから使う）。
- description は Claude が自動委譲の判断に使うため、具体的に書くほど効く。

### 段階導入のすすめ（一人プロジェクトの場合）
3エージェントを最初から全部使うより、まず explorer(haiku) だけ入れて
効果を体感 → 効くなら implementer を足す、が複雑さの割に合うか見極めやすい。

## 4. セッションプロンプト集

### 4-1. セッション再開（最頻出・これだけで始まる）
```
docs/STATUS.md を読んで、「次の一手」の1番から進めてください。
着手前に今日の作業ゴールを1〜3行で確認してください。
```

### 4-2. 新機能・新フェーズの開始
```
新しいフェーズに入ります: 「（例）Prophetでレビューポジ率の時系列予測」

まず実装せずに以下を提示してください:
1. 方針（アプローチの選択肢が複数あれば比較して推奨を明示）
2. 影響するファイル・新規作成するファイルの一覧
3. 作業を3〜5ステップに分割した計画（各ステップは1セッションで完了する粒度）

私が承認したら、ステップ1のみ着手してください。
```

### 4-3. 設計レビュー（Opusの得意分野に集中させる）
```
（対象ファイル/設計）についてレビューしてください。観点:
1. 設計上の欠陥・見落としているエッジケース
2. 拡張性（次フェーズ「○○」を見据えて問題ないか）
3. 過剰設計になっている箇所（シンプルにできる所）

コードの書き直しは不要です。指摘と理由のみ、重要度順に。
```

### 4-4. デバッグ
```
（エラー/症状の説明。エラーメッセージは要点のみ貼る）

まず原因の仮説を確度順に2〜3個挙げ、それぞれの検証方法を提示してください。
検証が必要な調査はexplorerサブエージェントを使ってください。
原因が確定するまで修正コードは書かないでください。
```

### 4-5. セッション終了（毎回必須）
```
今日はここまでにします。docs/STATUS.md を更新してください:
- 前回やったこと / 次の一手 / 未解決の問題
設計判断があった場合は docs/decisions.md にも追記してください。
コミット・プッシュ漏れがないかも確認してください。
```

### 4-6. コンテキストが重くなってきたとき
```
このセッションのここまでの作業内容と決定事項を docs/STATUS.md に反映してください。
反映後、新しいセッションで続きをやります。
```
（→ /clear または新セッションで 4-1 から再開。長いセッションを引きずるより安い）

---

## 5. 運用のコツ（人間側）

- **1セッション1目的**。目的を跨ぐならSTATUS.mdを更新して新セッションへ。Opusの長いコンテキストは高くつく
- 計画立案には**Planモード**（Shift+Tab）を使う。実装前の思考にファイル編集権限は不要
- `/cost` をたまに確認し、サブエージェント委譲が機能しているか見る
- STATUS.mdが肥大化してきたら、古い内容はdecisions.mdやWikiへ移して「現在地」だけに保つ
- 規約が増えてきたら CLAUDE.md に足さず、`.claude/rules/` にトピック別ファイルを作る
  （paths でスコープを絞れば常時ロードのコストがかからない）
- Claudeに同じ指摘を2回したら、その場で `#` から始めて即追記する
- このキット自体も自分のプロジェクトに合わせて削る・足す
