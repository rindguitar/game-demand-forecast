# CLAUDE.md

このファイルはClaude Codeがこのリポジトリで作業する際のガイドです。
**このファイルは薄く保つこと。** 詳細は `docs/` と `.claude/rules/` に置き、必要時のみ参照する（末尾の「参照ドキュメント」参照）。

## プロジェクト概要

Steam APIからレビュー・プレイヤー数・タグ等を収集し、NLP（感情分析・トピック抽出）と時系列分析でプレイヤー需要を予測するMLプロジェクト。パイプライン: データ収集 → NLP → 時系列予測 → 統合・ランキング。

### 技術スタック
- Python 3.10（Docker / WSL2 Ubuntu 22.04・CUDA 11.8）
- 主要ライブラリ: PyTorch 2.7.1+cu118 / Transformers・BERTopic・DistilBERT / Prophet→LSTM（時系列）
- データソース: Steam API
- GPU: NVIDIA GTX 1060 6GB（VRAM制約からbatch_size上限に注意）

## セッションプロトコル（最重要）

### 開始時
1. `docs/STATUS.md` を読む（現在地・次の一手・未解決事項）
2. 生成前に**今日のゴールを1-3行で確認**してユーザーと認識を合わせる

### 終了時
1. `docs/STATUS.md` を更新する（何をやった / 何が残った / 次に何をすべきか / 詰まった点）
2. 設計判断があれば `docs/decisions.md` に追記する
3. コミット・プッシュ漏れがないか確認する

## モデル運用ルール（コスト規律）

メインセッションの役割は**設計・レビュー・難所の実装のみ**。以下は必ずサブエージェントに委譲する:

| 作業 | 担当 | 理由 |
|---|---|---|
| コードベース探索・grep・ファイル特定 | explorer（haiku） | 安価・高速で十分 |
| テスト実行・結果要約 | test-runner（haiku） | 出力が長いだけの作業 |
| 実装（設計書に従うだけのコード） | implementer（sonnet） | 実装力は十分 |
| 設計・アーキテクチャ判断・コードレビュー | メイン（本体） | 判断が必要 |
| サブエージェントが2回失敗したタスク | メイン（本体） | エスカレーション |

- 大きなファイル（データCSV・ログ・モデル出力）を**メインのコンテキストに直接読み込まない**。必要ならサブエージェントに要約させる。
- 同じファイルを繰り返し読み直さない。一度読んだ内容は会話内の記憶を使う。
- IMPORTANT: 導入直後に一度 `/cost` を確認し、探索・テストが実際にhaiku/sonnetへ流れているか検証する（ルーティング不具合の既知報告あり）。

## 実装ワークフロー（説明可能性の担保）

**実装前に方針を先に提示し、承認を得てから書く。**
1. 方針提示: 何をどう変えるかを箇条書き3-5行で（差分のイメージ）
2. 承認後に実装
3. テスト実行 → 成功確認 → コミット → プッシュ（この順序を崩さない）

理由: ユーザーは「自分で説明できないコードは採用しない」。方針を先に合わせることで、書き直しによるトークン浪費を防ぐ。

## 開発コマンド

makeコマンドを優先（直接 `docker compose` 等を叩かない）。
- 全コマンド一覧: `make help`
- テスト: `make test`
- Lint / Format: `make lint` / `make format`

### ⚠️ 時間のかかる操作はユーザーが実行
以下はClaude Codeで実行せず、実行が必要だとユーザーに通知する:
- Dockerイメージの再ビルド
- GPU長時間学習・大量データのダウンロード

## コーディング規約（常時適用の最小限）

- **処理の流れをコメントで説明する**: 関数・処理ブロックの冒頭を区切りにし「何をどうやるか」順で日本語コメントを書く（例: `# 1. レビューを日付でグループ化 → 2. 日次ポジ率を算出 → 3. 欠損日を補間`）。ユーザーがコードを読んで流れを追えることが目的。ただし自明な1行ごとのコメント（`i += 1  # 加算`等）は書かない。
- コメント・docstringは日本語（技術用語は英語のまま。例: accuracy, batch, pipeline）
- 変数名・関数名は英語（Pythonの慣例に従う）
- ハードコード禁止。設定値は引数・環境変数・設定ファイルへ
- トピック別の詳細規約は `.claude/rules/` にある（該当ファイル編集時に自動ロード）

## Git規約

- IMPORTANT: `main` への直接コミット禁止。コード・ドキュメントの変更は必ず作業ブランチを切ってから行う。
- 作業開始時に `git branch --show-current` で現在のブランチを確認する。`main` にいたら `feature/<内容>` または `fix/<内容>` を切る（例: `feature/prophet-baseline`）。
- 例外: `docs/STATUS.md` の更新のみ、現在の作業ブランチ上でそのままコミットしてよい。
- 新規ブランチは作業開始前にリモートへプッシュ。
- コミットメッセージ: `feat:` / `fix:` / `docs:` / `refactor:` / `chore:`
- ブランチ名に迷う場合（typo修正等の微小変更）はユーザーに確認する。

## セキュリティ

- `.env` は `.claudeignore` で保護済み。APIキーの設定はユーザーが手動で行う（依頼しても行わない）。
- コミット前に実際の秘密鍵・トークン等が含まれていないか確認。疑わしければコミットしない。
